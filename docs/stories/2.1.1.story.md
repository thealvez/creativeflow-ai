## 🎯 Story Summary

Design and implement OAuth 2.0 authentication flows for Google and GitHub, including authorization endpoints, callback handlers, token management, and user creation/login logic.

---

## 👤 User Story

**As a** user
**I want** to login with my Google or GitHub account
**So that** I can access CreativeFlow AI without creating a new password.

## ✅ Acceptance Criteria

### AC-1: Google OAuth Flow
```
✓ POST /api/auth/google endpoint accepts redirect_uri parameter
✓ Returns authorization_url (Google consent screen)
✓ State parameter generated and stored (CSRF protection)
✓ GET /api/auth/google/callback handles auth code
✓ Exchanges code for access token from Google
✓ Retrieves user profile (email, name, avatar)
✓ Creates user if not exists, updates if exists
✓ Returns JWT token + refresh token to frontend
✓ Frontend redirected to /dashboard with token in httpOnly cookie
```

### AC-2: GitHub OAuth Flow
```
✓ POST /api/auth/github endpoint accepts redirect_uri parameter
✓ Returns authorization_url (GitHub consent screen)
✓ State parameter generated and stored
✓ GET /api/auth/github/callback handles auth code
✓ Exchanges code for access token from GitHub
✓ Retrieves user profile (login, name, avatar_url)
✓ Creates user if not exists, updates if exists
✓ Returns JWT token + refresh token
✓ Frontend redirected to /dashboard with token in httpOnly cookie
```

### AC-3: Token Management
```
✓ JWT token generated with:
  - sub (user_id)
  - email
  - iat (issued_at)
  - exp (24 hours from now)
  - iss (creativeflow.local)

✓ Refresh token stored in Redis:
  - Key: refresh_token:{userId}
  - TTL: 7 days
  - Token rotated on use (new refresh token issued)

✓ Token rotation logic:
  - Old refresh token invalidated after rotation
  - New refresh token returned to frontend
  - Prevents refresh token reuse attacks
```

### AC-4: User Model & Database
```
✓ User record created with:
  - id (UUID)
  - email (unique)
  - name
  - avatar_url
  - oauth_provider (google or github)
  - oauth_id (provider's user ID)
  - subscription_tier (default: free)
  - created_at, updated_at

✓ oauth_integrations table:
  - Tracks multiple OAuth connections per user
  - Stores access_token (encrypted), refresh_token (encrypted)
  - Stores token expiry for token refresh
```

### AC-5: Error Handling
```
✓ Invalid state parameter → 400 Bad Request
✓ Expired authorization code → 400 Bad Request
✓ Google/GitHub API timeout → 503 Service Unavailable (with retry)
✓ Database error → 500 Internal Server Error (with logging)
✓ All errors return structured JSON response
✓ No sensitive info exposed in error messages
```

### AC-6: Security Requirements
```
✓ HTTPS enforced (redirect_uri must be https in prod)
✓ State parameter validated (prevents CSRF)
✓ Tokens stored in httpOnly cookies (not localStorage)
✓ Token never exposed in URL or logs
✓ Refresh token rotation implemented
✓ PKCE flow supported (optional for server-side apps)
✓ Cookie SameSite=Strict
```

### AC-7: Frontend Integration
```
✓ Login form with "Google" and "GitHub" buttons
✓ Clicking button opens OAuth consent screen
✓ After consent, redirects to /dashboard
✓ Token stored in httpOnly cookie
✓ API requests include Authorization: Bearer {token}
✓ On token expiry, auto-refresh token silently
✓ Logout clears token from cookie
```

### AC-8: Testing & Documentation
```
✓ Unit tests for OAuth flow logic
✓ Integration tests for Google/GitHub callbacks
✓ Token rotation tested
✓ Error scenarios tested
✓ OAuth setup documented:
  - How to create Google OAuth app
  - How to create GitHub OAuth app
  - Required scopes explained
  - Redirect URIs documented
```

## 🛠️ Technical Details

### **Google OAuth Setup**
1. Create OAuth 2.0 credentials in Google Cloud Console
2. Set authorized redirect URIs: `{backend_url}/api/auth/google/callback`
3. Copy Client ID and Client Secret to .env
4. Use scopes: `openid email profile`

### **GitHub OAuth Setup**
1. Create OAuth app in GitHub Settings → Developer settings
2. Set Authorization callback URL: `{backend_url}/api/auth/github/callback`
3. Copy Client ID and Client Secret to .env
4. Use scopes: `user:email`

### **JWT Implementation**
```javascript
// Header
{ alg: 'RS256', typ: 'JWT', kid: 'key_id' }

// Payload
{
  sub: 'user_123',
  email: 'user@example.com',
  name: 'Sofia',
  iat: 1678900000,
  exp: 1678986400,
  iss: 'creativeflow.local'
}

// Signature
RSA-SHA256(base64(header) + '.' + base64(payload), private_key)
```

### **Refresh Token Flow**
```
1. User authenticates → Get access_token (24h) + refresh_token (7d)
2. Access token expires → Frontend calls POST /api/auth/refresh
3. Backend validates refresh_token in Redis
4. Issues new access_token + new refresh_token
5. Invalidates old refresh_token
6. Frontend updates cookie with new token
```
